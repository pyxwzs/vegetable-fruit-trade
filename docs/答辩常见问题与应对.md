# 毕业答辩 — 常见问题与应对角度（果蔬贸易系统）

本文从**答辩老师常见提问习惯**出发，结合本系统（前后端分离、进销存、JWT 权限、采购/销售/库存/财务等）整理**可能的问题与回答要点**。正式答辩时请用自己的话展开，并结合你论文里的图、表、用例编号作答。

---

## 一、选题与意义类

| 可能的问题 | 回答要点（抓关键词即可） |
|------------|-------------------------|
| 为什么选「果蔬贸易」这个题目？ | 行业周转快、对**库存保质期**和**进销衔接**敏感；中小企业信息化程度参差不齐，用统一系统可降低差错、便于追溯。 |
| 你的系统和市面上进销存软件有什么差别？ | 说明**范围边界**：本课题侧重教学/原型级完整链路（Web 管理端、角色权限、采购销售与库存联动、部分分析/财务视角），不宣称与商业 ERP 全面对标；若有特色（如 FIFO、授信校验）可点一句。 |
| 实际调研过吗？用户是谁？ | 如实回答：若有访谈/问卷写具体；若为**文献+类比推理**，说明依据行业报告或岗位观察，并承认样本局限。 |

## 二、需求与分析类

| 可能的问题 | 回答要点 |
|------------|----------|
| 核心用例是哪几个？画一下用例图。 | **登录与权限**、**商品/主数据维护**、**采购下单—审核—发货—收货入库**、**销售下单—审核—发货出库**、**库存查询/预警**、（可选）**退货/财务汇总**。用例图与论文保持一致。 |
| 谁可以做什么？权限怎么分的？ | **基于角色的访问控制（RBAC）**；权限以字符串编码（如 `inventory:inbound`），接口上用 `@PreAuthorize`；论文里可对照权限表或菜单说明。 |
| 有没有考虑并发？两个人同时改库存怎么办？ | **事务**：关键写操作在 Service 层 `@Transactional`；高并发下仅靠应用层事务不够，可补充**乐观锁/悲观锁/库存流水表**作为展望（若论文未实现要诚实说明）。 |

## 三、总体架构与技术选型类

| 可能的问题 | 回答要点 |
|------------|----------|
| 为什么前后端分离？ | 职责清晰、前后端可独立演进；前端专注交互与展示，后端专注业务与数据；部署上静态资源与 API 可分开扩展。 |
| 为什么用 Spring Boot？ | 约定优于配置、生态成熟（Security、JPA、Validation）、适合快速构建 REST 服务，与课程/岗位技能衔接。 |
| 为什么用 Vue？ | 组件化、响应式数据流、与 Element Plus 适合管理后台；与 Spring Boot 通过 JSON/HTTP 协作清晰。 |
| 为什么用 JWT 而不是 Session？ | **无状态**、易横向扩展；注意说明 refresh 机制与 XSS/泄露风险及你的缓解方式（HTTPS、HttpOnly 若未用要承认是改进点）。 |

## 四、数据库与设计类

| 可能的问题 | 回答要点 |
|------------|----------|
| 核心表有哪些？ER 图怎么画的？ | 围绕**用户/角色/权限**、**商品/分类**、**供应商/客户/仓库**、**采购单及明细**、**销售单及明细**、**库存（含批次等字段若论文有）**、（可选）**退货/操作日志**说明实体关系；与论文 ER 图一致。 |
| 用 JPA 的 ddl-auto 是什么？生产能用吗？ | 开发常用 `update` 自动演进；生产一般用**迁移脚本（Flyway/Liquibase）**或显式 DDL，避免误改表结构——可作为**不足与改进**。 |
| 金额用什么类型？ | **`BigDecimal`**，避免 `double` 金融误差；订单金额计算在服务端完成。 |

## 五、安全与认证类（高频）

| 可能的问题 | 回答要点 |
|------------|----------|
| 登录流程是怎么实现的？ | 用户名密码经 `AuthenticationManager` 认证；通过后签发 **access + refresh**；可选 **MFA** 时先返回挑战再换 token（与 `UserService.login` 一致即可）。 |
| JWT 存在哪里？安全吗？ | 当前前端多在 **localStorage**；说明**优点**（实现简单）与**风险**（XSS 窃取）；可答若改进可用 **HttpOnly Cookie + CSRF 防护** 等。 |
| 密码怎么存的？ | **BCrypt** 哈希，不明文落库；传输层建议 **HTTPS**（部署环境说明）。 |
| 接口怎么防止越权？ | Spring Security 全局需认证 + 方法级 **`@PreAuthorize`**；管理员「代查数据视角」若论文有写，强调**仅存服务端、JWT 不含可篡改视角**。 |

## 六、业务与库存类（易深挖）

| 可能的问题 | 回答要点 |
|------------|----------|
| 入库、出库逻辑是什么？ | 按**商品 + 仓库 + 批次**定位库存行；入库累加数量与可用量；出库前校验**可用量 ≥ 出库量**，否则抛业务异常。 |
| 什么是 FIFO？为什么销售发货用 FIFO？ | **先入库先出库**，利于保质期管理与成本近似（若论文按批次管理可结合说明）；实现上按可用库存排序后循环扣减直至满足数量。 |
| 采购收货后库存怎么变？ | 订单状态到「已发货」后再「收货」时，遍历明细调用 **入库服务**，写入对应仓库（实现里若有默认仓库 ID，说明是简化假设，可写进展望：按明细指定仓库）。 |
| 销售授信额度怎么控制？ | 创建销售单前汇总**未结清占用 + 本单金额**与客户**授信额度**比较，超限拒绝下单（对应 `SalesService.createSalesOrder` 思路）。 |
| 退货流程和业务单据关系？ | 区分**仅申请记录**与**审批后联动库存/资金**；结合你论文中退货/返款状态机简述，避免和代码实现细节矛盾。 |

## 七、非功能与工程类

| 可能的问题 | 回答要点 |
|------------|----------|
| 异常怎么处理？前端怎么提示？ | 业务异常返回统一结构（如 `ApiResponse` + message）；全局异常处理器（若有）统一转换；前端 Axios 拦截器提示错误信息。 |
| 操作日志怎么做的？ | **AOP 环绕** Controller（及 `@Loggable`），记录 IP、URL、用户、参数摘要、结果/异常、耗时；异步或同步保存视实现说明。 |
| 文件上传怎么做？ | **Multipart** 接收，**MinIO** 存对象，库中存元数据；禁用时要有明确提示（与 `FileService` 一致）。 |
| 有没有做接口文档？ | 如实：Swagger/Apifox/Word 附录；若没有，说**按 Controller + DTO 维护**并作为改进点。 |

## 十、批判性提问（要有心理准备）

| 可能的问题 | 回答要点 |
|------------|----------|
| 你的创新点在哪里？ | 不必夸大「首创」：可说**在果蔬场景下把保质期/临期预警与进销存流程结合**、或**权限+业务状态机较完整**、或**财务汇总与业务单据联动**等——与论文「创新点」小节一致。 |
| 若库存数据被恶意篡改怎么办？ | 业务只能通过**受控接口**修改；数据库权限与备份；更进一步可谈**审计日志、只读副本、签名校验**——承认当前是管理端信任模型。 |
| 为什么默认仓库写死为 1L？ | 如实：**课程设计简化**；扩展为「明细行带仓库」或「用户默认仓」即可，老师问的是你是否**意识到假设边界**。 |

| 参考文献有多少篇？近五年的比例？ | 如实统计；说明**经典文献**（如软件工程教材）与**近五年技术博客/官方文档**的搭配。 |

### 11.4 接口与前后端协作

| 可能的问题 | 回答要点 |
|------------|----------|
| REST 接口怎么设计的？资源名用什么？ | 名词复数、HTTP 动词语义；统一 **`/api` 前缀**与 `ApiResponse` 包装。 |
| 跨域怎么解决的？生产环境呢？ | 开发：**devServer 代理**；生产：**同源部署**或网关 CORS；避免 `*` 配 credentials 的坑可提一句。 |

| 401 与 403 区别？你们系统怎么用的？ | 401 未认证/令牌失效；403 已认证无权限；结合 Axios 刷新与 `ElMessage` 说明。 |

### 11.5 数据一致性、事务与幂等

| 可能的问题 | 回答要点 |
|------------|----------|
| 哪些操作必须在同一事务里？ | 典型：**收货 + 多行入库**、**发货 + FIFO 扣减**；说明 `@Transactional` 边界。 |
| 若扣库存中途失败会怎样？ | **整单回滚**；用户看到失败提示，数据不脏读。 |
| 重复提交同一按钮会怎样？ | 如实：若**未做幂等 token**，可能重复单据——作为改进点；可提前端防抖或后端唯一键。 |
| 删除商品/客户失败是什么原因？（若已实现） | **外键约束**：先提示关联库存或订单；或说明全局异常转友好文案。 |

### 11.6 数据库进阶

| 可能的问题 | 回答要点 |
|------------|----------|
| 建了哪些索引？为什么？ | 如订单号唯一、常用查询字段；没有索引的诚实说**数据量小未优化**，展望 EXPLAIN。 |
| 为什么用 MySQL 不用 PostgreSQL/SQLite？ | **生态与课程衔接**、部署习惯；承认 PG 在部分特性上更强也无妨。 |
| N+1 查询有没有遇到？ | 若有 **EntityGraph / JOIN FETCH** 可举例（如采购单列表）；没有则说明列表字段简单。 |
| 软删除与物理删除你怎么选的？ | 主数据倾向**停用**；若物理删要清外键——与当前实现一致即可。 |

### 11.7 安全进阶（除 JWT 外）

| 可能的问题 | 回答要点 |
|------------|----------|
| SQL 注入怎么防？ | **JPA 参数绑定**、禁止拼接 SQL；搜索关键字若原生 SQL 需说明转义或预编译。 |
| XSS、CSRF 考虑过吗？ | 富文本若存在说明**消毒或白名单**；JWT 在 Header 时 CSRF 风险相对低；Cookie 方案则要 CSRF Token。 |
| 敏感配置放哪？能提交 Git 吗？ | **环境变量 / 配置中心**；`application.yml` 中密码应**脱敏、不进库**——承认曾疏忽可答已改。 |
| 接口限流、防暴力破解？ | 若未实现：诚实 + **登录失败锁定、验证码、网关限流**作展望。 |

### 11.15 对比与批判思维

| 可能的问题 | 回答要点 |
|------------|----------|
| 若用微服务重写，会怎么拆？ | **用户/商品/订单/库存/财务**等限界上下文；说明毕设用单体更合适的理由（成本、复杂度）。 |
| 与 Excel 记账比优势在哪？ | **并发协作、权限、追溯、库存准确**；承认单人场景 Excel 仍够用，体现思辨。 |

### 11.16 个人与团队（若多人合作）

| 可能的问题 | 回答要点 |
|------------|----------|
| 你负责哪一块？接口怎么对接的？ | **具体模块** + 前后端约定（路径、字段、错误码）；避免与队友陈述冲突。 |
| 遇到分歧怎么解决？ | 导师拍板 / 文档先行 / 接口契约；体现沟通。 |

### 11.17 临场压力与「挖坑」题

| 可能的问题 | 回答要点 |
|------------|----------|
| 你这段代码明显有 bug，你怎么看？ | 先冷静：**确认是否理解题意**；若是真问题，感谢指出，说明**修复思路或已记录**。 |
| 这个功能演示失败了。 | **检查网络/服务/数据**；可切换备用录屏；态度重于当场修好。 |
| 用一句话概括你的毕设。 | 提前背一句：**为谁、解决什么问题、用什么技术、达成什么效果**。 |

## 十二、答辩前自检清单（1 分钟过一遍）

1. 论文目录与**系统截图、ER 图、用例图**是否一致。  
2. 能口述：**三层架构** + **JWT 从登录到请求头再到过滤器** 的路径。  
3. 能口述：**采购收货 → 入库**、**销售发货 → FIFO 出库** 各一步。  
4. 准备好一句**诚实的话**：未实现或简化的部分及改进方向。  
5. 演示环境：**后端端口、context-path、前端代理、测试账号**提前启动验证一遍。
